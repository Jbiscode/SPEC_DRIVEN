# Part 6: 실무에서 쓰는 흐름

## Spec-Driven 워크플로우

```
1. SPEC 작성 (사람)
      ↓
2. AI가 TODO 작성
      ↓
3. AI가 코드 작성
      ↓
4. 개발자가 코드 리뷰 (또는 AI로 리뷰 보조)
```

- **개발자**는 **설계(Spec)** 와 **리뷰**에 집중  
- **Spec이 source of truth** → "이 Spec대로 구현됐는가?"로 검증

## Before / After 예시

같은 요청이라도 **규칙 유무**에 따라 결과가 달라집니다.

**요청**: "로그인 실패 시 사용자에게 메시지 보여주는 코드 만들어줘."

| | 규칙 없이 | Rules 있이 (예: "에러 메시지는 사용자 노출용/로그용 구분", "한국어 메시지") |
|--|-----------|-----------------------------------------------------------------------------|
| **결과** | 메시지 문구·형식 제각각, 로그에 그대로 찍을지 말지 불명확 | "사용자용: 로그인에 실패했습니다. 다시 시도해 주세요." / "로그용: Login failed, reason=…" 식으로 구분된 형태로 일관되게 나옴 |
| **리뷰** | "이거 우리 팀 스타일 아님" 등 주관적 피드백 증가 | "Rule O 준수했는지"로 객관적으로 확인 가능 |

이렇게 **한두 가지 Rule만 있어도** 출력 일관성과 리뷰 비용이 달라집니다.

## Spec 최소 템플릿

AI에게 구현을 맡기기 전에, **최소한으로** 적어 둘 만한 항목입니다. (기능 한 건 기준.)

```markdown
## 기능 이름
- **목적**: (한 줄)
- **입력**: (파라미터/이벤트/API 요청 등)
- **출력**: (반환값/응답/화면 동작 등)
- **예외/에러**: (어떤 경우에 무엇을 할지)
- **참고**: (기존 코드/문서 링크)
```

실제로는 "입력·출력·예외"만 있어도 Spec으로서 역할을 합니다. 팀에서 API 명세나 이슈 템플릿이 있다면 그걸 Spec으로 쓰고, "이 Spec 기준으로 구현해줘"라고 요청하면 됩니다.

**이걸 내 도구에 적용하려면 어디에 넣어야 할지, 그리고 피해야 할 패턴을 보자.**
