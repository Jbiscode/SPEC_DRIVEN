# Part 8: 팀과 협업·면접

## 프로젝트 관점: 일정·리스크·협업

**일정 관점**
- **Rules/Skills가 없으면**: 사람마다 AI 사용 방식이 달라 결과물 편차가 크고, 재작업이 늘어 일정이 흔들림
- **있으면**: 온보딩 빨라짐, 반복 업무 시간 단축, 결과물 형식 표준화, 리뷰 비용 감소

**리스크 관점**  
AI를 잘못 쓰면 말투/형식 제각각, 사실 검증 누락, 민감정보 처리 실수, 팀 규칙 미반영 등이 생깁니다.  
**Rules**는 이런 운영 리스크를 낮추고, **Skills**는 업무 누락과 품질 편차를 줄여줍니다.

**협업 관점**  
단순히 코드를 AI에게 맡기면 **일관성이 없고**, 개인 프로젝트가 아니라 **팀과 협업**해야 하니까 같은 팀원끼리 실제로 개발할 때 **충돌 없이**, **생산성을 어떻게 올릴지**를 많이 고민하게 됩니다.  
개인이 "나만의 프롬프트"로 잘하는 건 **팀 자산이 아닙니다**.  
Rules와 Skills로 정리하면 **팀 공용 자산**이 되어, 그 고민을 줄일 수 있습니다.

## 개인 vs 팀 자산 & 면접 답변

- 개인이 "나만의 프롬프트"로 잘하는 것은 **팀 자산이 아니다**.
- **Rules와 Skills로 정리하면 팀 공용 자산**이 된다.

**면접에서 강한 답변**

| 약한 답변 | 강한 답변 |
|-----------|-----------|
| "AI 활용 경험이 있다" | "AI 사용 방식을 **팀이 재사용 가능하게 구조화**해봤다" |

"어떤 도구를 썼다"보다 **"어떻게 규칙·스킬로 정리해 팀이 같이 쓰게 했는지"**를 구체적으로 말하는 것이 좋습니다.

## 면접에서 나오는 질문 & 실제 할 만한 답변

### Q1. "개발할 때 얼마나 AI 써봤어요?" / "AI 활용 경험이 있나요?"

**짧게 (30초~1분)**  
> "일상적으로 씁니다. 코드 작성·리팩토링, API 스펙 정리, PR 설명 작성, 테스트 케이스나 문서 초안을 만들 때 AI를 쓰고, **단순히 질문만 하는 게 아니라 요구사항을 Spec으로 먼저 정리한 뒤 그걸 기준으로 작업**하도록 했어요. 그래서 결과물 품질이 들쭉날쭉하지 않게 맞추려고 했습니다."

**조금 더 구체적으로**  
> "단순히 코드를 AI에게 맡기면 일관성이 없고, 개인 프로젝트가 아니라 팀이랑 협업해야 하니까 **같은 팀원끼리 실제로 개발할 때 충돌 없이, 생산성을 어떻게 올릴지** 많이 고민했어요. 그 과정에서 개인 작업만큼 팀에서 쓸 때도 신경 썼습니다. 예를 들어 **Rules**로 '코드 수정 전 영향 범위 먼저 설명', '팀 코딩 스타일 준수' 같은 걸 두고, **Skills**로 PR 요약 템플릿이나 코드 리뷰 체크리스트처럼 반복되는 작업을 묶어뒀어요. 그래서 '어떤 도구 썼다'보다 **어떻게 규칙·스킬로 정리해 팀이 같이 쓸 수 있게 했는지**를 말할 수 있습니다."

---

### Q2. "AI 어떻게 활용했어요?" / "구체적으로 어떤 식으로 쓰나요?"

**실제 면접에서 할 만한 답변 (1분 내외)**  
> "세 가지 축으로 정리해서 씁니다.  
>  
> **첫째, Spec과 Context입니다.** 요구사항을 명세로 먼저 적고, 그걸 AI에게 주면서 '이걸 기준으로 구현해 달라'라고 합니다. 그래서 **AI가 우리가 원하는 문맥을 제대로 알고 답하도록** Context를 설계하는 편이에요.  
>  
> **둘째, Rules와 Skills입니다.** Rules는 '항상 한국어로', '코드 변경 시 영향 범위 먼저 설명', '보안 정보 마스킹' 같은 **전역 원칙**이고, Skills는 API 클라이언트 생성, PR 요약, 코드 리뷰 체크리스트처럼 **자주 쓰는 작업을 재사용 가능하게 묶어둔 것**이에요. 이렇게 하면 품질 편차가 줄고, 팀이 같이 쓸 수 있는 자산이 됩니다.  
>  
> **셋째, 개인적으로는** 자주 쓰는 Rules·Skills를 프로젝트마다 모아두었다가, 비슷한 기능이나 다음 프로젝트에서 다시 꺼내 쓰면서 **나만의 워크플로로 쌓아가는 중**입니다."

**따라서 강조할 포인트**  
- "단순히 코드를 AI에게 맡기면 **일관성이 없고**, 팀 협업 시 **충돌 없이 생산성을 올리려고** 고민했다."  
- "그래서 단순히 프롬프트만 잘 쓰는 게 아니라, **Spec·Context·Rules·Skills로 구조화**해서 쓴다."  
- "**팀이 재사용할 수 있게** 정리해뒀다."  
- "**반복 작업은 규칙·스킬로 고정**해서 품질과 일정을 잡았다."

---

## 개인 자산으로 쌓기: 나만의 무기

- 프로젝트를 하면서 **자주 쓰는 Rules·Skills를 잘 모아두면**, 다음번 비슷한 프로젝트나 기능을 만들 때 그대로 꺼내 쓸 수 있다.
- 이걸 반복해서 쌓아두면 **실무 능력이 올라가고**, **나만의 무기**가 된다.
- 이런 경험을 만들려면 **많이 만들어보는 것**이 중요하다. 여러 프로젝트·기능을 해보면서 “이건 따로 규칙/스킬로 남겨두자”고 정리하는 습관을 들이면 된다.
