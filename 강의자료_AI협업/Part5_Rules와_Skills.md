# Part 5: Rules & Skills

## 비유로 이해하기

| 구분 | 의미 |
|------|------|
| **프롬프트** | 그때그때 주는 지시 |
| **Rules** | 회사 업무 매뉴얼 |
| **Skills** | 자주 쓰는 표준 업무 프로세스 |

## Rules란?

AI가 답하거나 작업할 때 **항상 따라야 하는 행동 원칙**입니다. 즉, AI의 **행동 기준선**입니다.

**예시**
- 항상 한국어로 답하라
- 코드 변경 시 영향 범위를 먼저 설명하라
- 코드 수정 전 테스트 포인트를 먼저 제시하라
- 보안 관련 정보는 외부로 내보내지 마라
- 코딩 스타일, API 출력 포맷, 네이밍 규칙, 개인정보 처리 원칙

**왜 중요한가**  
AI는 유연해서 같은 질문에도 답변 방식이 흔들릴 수 있습니다. Rules를 두면 결과물이 **일관되고 예측 가능**해집니다.

> 면접에서: "AI를 쓸 때 단순히 잘 물어보는 것보다, 모델이 항상 따라야 할 규칙을 설계해서 품질 편차를 줄이는 게 중요하다."

## Skills란?

특정 업무를 **반복 가능하게 가르쳐두는 작은 기능 묶음**입니다.  
매번 길게 설명하지 않고, **지침 + 참고자료 + 필요하면 스크립트**까지 묶어둔 것 = AI용 **업무 템플릿 + 작업 절차 자동화 패키지**에 가깝습니다.

**예시**
- API 스펙에서 클라이언트 코드 생성하는 skill
- 코드 리뷰 체크리스트대로 검토해 주는 skill
- 로그/에러 메시지를 팀 포맷으로 정리하는 skill
- PR 설명 템플릿에 맞춰 요약해 주는 skill
- 이력서용 문장 다듬기, 리팩토링 제안 요약 등

**구조를 단순하게 보면**
- 이 skill이 **언제 쓰이는지**
- **어떤 순서**로 작업하는지
- **어떤 형식**으로 결과를 내야 하는지
- 참고해야 할 **문서/스크립트**가 있는지  

이런 걸 묶어두는 겁니다.

**이게 바로 스킬의 큰 장점(Kick!) 중 하나입니다.**  
예를 들어, 반복적인 작업이 발생할 때 AI가 매번 답변을 만들어 토큰을 소비하는 대신, **스킬에 미리 스크립트(.sh, .py 등)를 연결해두면 해당 작업이 트리거됐을 때 직접 실행**할 수 있습니다.  
즉, 사람이 별도로 설명하지 않아도 스킬이 알아서 스크립트를 실행하고 결과만 받아오기 때문에 **AI가 처리할 몫과 토큰 사용량을 확 줄일 수 있다는 게 핵심 Kick**입니다.  
(실제로 커서 등 도구마다 “스킬 실행 시 지정된 스크립트를 자동으로 돌리는” 기능을 공식 지원하기도 합니다.)

## Rules와 Skills 한 줄 구분

| | Rules | Skills |
|---|--------|--------|
| **역할** | 전역적인 운영 원칙 | 특정 작업을 잘 수행하는 재사용 모듈 |
| **예** | "코드 스타일 준수", "근거 없는 추정 금지", "보안 정보 마스킹" | "API 클라이언트 생성 skill", "코드 리뷰 체크리스트 skill", "PR 요약 skill" |
| **한 줄** | **어떻게 행동할지** | **어떤 일을 어떻게 처리할지** |

> "저는 AI를 단발성 질문 도구로만 쓰지 않고, 반복 업무는 규칙과 템플릿으로 구조화해서 재사용하려고 했습니다. 예를 들어 API 출력 포맷 규칙을 정하고, 자주 쓰는 작업은 skill처럼 표준화해서 품질 편차를 줄이는 방식으로 활용했습니다."

## 좋은 프롬프트와 Rules의 차이

| 구분 | 프롬프트 | Rules |
|------|----------|--------|
| **범위** | 한 번의 대화/요청에 적용 | **항상** 적용 (세션/프로젝트 단위) |
| **재사용** | 복붙·수정 필요 | 저장해 두면 **자동 적용** |
| **소유** | 개인용이 되기 쉬움 | **팀 공용 자산**으로 만들기 좋음 |
| **역할** | "이번에만 이렇게 해줘" | "우리 팀은 항상 이렇게 한다" |

- **좋은 프롬프트**: 그때그때 맥락을 잘 주어 한 번의 결과물 품질을 높임  
- **Rules**: 그런 "항상 지켜야 할 것"을 **고정**해 두어, 프롬프트를 매번 길게 쓰지 않아도 되게 함

**이 구조를 실무에서는 어떻게 쓰는지, 워크플로우부터 보자.**
