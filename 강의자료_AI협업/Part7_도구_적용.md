# Part 7: 도구에 적용하기

## 도구별 Rules/Skills 넣는 곳

개념만 알면 "어디에 어떻게 넣지?"에서 막힐 수 있습니다. 자주 쓰는 도구 기준으로 정리합니다.

| 도구 | Rules 넣는 곳 | Skills 넣는 곳 |
|------|----------------|----------------|
| **Cursor** | `.cursor/rules` 폴더의 `.mdc` 파일, 또는 프로젝트 Rules 설정 | `.cursor/skills/` 또는 Cursor 설정에서 Skills 라이브러리 |
| **ChatGPT** | 설정 → Custom instructions (프로젝트/대화 공통 원칙) | GPT에서 "이 작업용 instruction 만들어줘"로 저장 후 재사용 |
| **Claude (Codex 등)** | 프로젝트별 지침 파일 또는 시스템 프롬프트 영역 | 동일 도구의 "재사용 가능 지침" 기능이 있다면 거기에 저장 |

**실무 팁**
- Rules는 **짧고 검증 가능한 문장**으로. 한 파일에 다 넣기보다 **역할별로 파일 분리**(코드스타일.mdc, 보안.mdc 등)하면 수정이 쉽습니다.
- Skills는 **입력 형식 → 출력 형식**이 명확한 작업부터 만드는 것이 좋습니다.

## 피해야 할 패턴 & 실무 팁

**자주 하는 실수**
- **Spec 없이 "이거 구현해줘"부터 요청**  
  → 범위가 흔들리고, 리뷰 기준이 없어집니다. 최소한 "입력/출력/예외"만이라도 적어 두고 요청하세요.
- **Rules를 너무 많이·길게**  
  → 모델이 전부 반영하지 못하거나 우선순위가 흐려질 수 있습니다. 핵심 3~5개부터 적용하고, 효과 보면 하나씩 추가하세요.
- **Context 없이 "우리 코드 스타일 맞춰줘"**  
  → 실제 코드베이스·컨벤션 문서를 함께 넣어야 합니다. "우리 프로젝트의 OO 파일 참고해서"처럼 **구체적인 참조**를 주세요.
- **Skill 하나에 너무 많은 단계**  
  → 한 Skill은 "한 가지 목표(예: PR 설명 생성)"에 맞추는 편이 재사용과 수정이 쉽습니다.

**실무 팁**
- **규칙은 "금지"보다 "이렇게 하라"**가 더 잘 지켜집니다.  
  예: "API 키를 코드에 넣지 마라"보다 "환경 변수 `API_KEY`를 사용하라".
- **팀에 공유할 때**: Rules/Skills를 문서로만 두지 말고, **실제 도구에 넣어 두고 버전 관리**(예: `.cursor/rules`를 repo에 포함)하면 온보딩이 빨라집니다.
- **리뷰 시**: "Spec에 있는가?" "정해 둔 rule을 위반했는가?"를 체크리스트로 두면 논의가 명확해집니다.

**팀 단위로 보면 일정·리스크·협업이 어떻게 달라지는지, 그리고 면접에서 어떻게 말하면 좋을지 정리하자.**
