# Part 9: 마무리 — FAQ·면접·요약·다음 단계

## FAQ: 팀에서 AI 결과물 편차를 줄이려면 어떤 rules가 필요한가?

**제안하는 rules 예시:**

1. **출력 포맷 규칙**  
   - API 응답 형식, 로그 포맷, 문서/주석 스타일을 팀에서 정해 두고 rule로 명시
2. **코드 스타일·컨벤션**  
   - 언어별 코딩 스타일, 네이밍, 파일/폴더 구조를 rule로 정의
3. **보안·개인정보**  
   - 비밀/키/개인정보를 코드·로그·주석에 넣지 않기, 마스킹 규칙 등
4. **리뷰 기준**  
   - "이 조건이면 반드시 수정" 같은 최소 기준을 rule로 정리
5. **문서화·Spec 준수**  
   - "Spec에 없는 동작은 구현하지 않는다", "변경 시 Spec 먼저 갱신" 등

팀마다 다르므로, **실제로 편차가 나는 부분**부터 rule로 잡는 것이 좋다.

## 스터디에서 같이 보면 좋은 질문

1. 좋은 프롬프트와 rules의 차이는 무엇인가?
2. 어떤 작업이 skill로 분리될 만큼 반복적인가?
3. 팀에서 AI 결과물 편차를 줄이려면 어떤 rules가 필요한가?
4. 개인 노하우를 팀 자산으로 바꾸려면 무엇을 문서화해야 하나?

## 면접 답변·키워드

**한 줄 정의**
- **Rules**: AI가 항상 지켜야 하는 공통 원칙
- **Skills**: 반복 업무를 잘하게 만드는 재사용 가능한 작업 모듈

**면접 답변 예시**  
> "저는 AI를 단발성 질문 도구로만 쓰지 않고, 반복 업무는 규칙과 템플릿으로 구조화해서 재사용하려고 했습니다. 예를 들어 API 출력 포맷 규칙을 정하고, 자주 쓰는 작업은 skill처럼 표준화해서 품질 편차를 줄이는 방식으로 활용했습니다."

**이 답변의 장점**: 단순 사용자처럼 안 보임, AI를 팀 생산성 관점에서 이해하는 사람처럼 보임, 실무형 사고를 보여줌.

**면접용으로 같이 묶으면 좋은 키워드 (4개)**
- **Spec Driven Development**: 구현 전에 명세를 명확히 만드는 방식
- **Context Engineering**: AI에게 어떤 문맥을 넣을지 설계하는 것. (프롬프트 엔지니어링에서 전환—요즘은 "어떤 컨텍스트를 주고 어떻게 파악하게 할지"가 더 중요.)
- **Rules**: AI가 항상 따를 원칙
- **Skills**: 반복 작업을 재사용 가능하게 만든 단위

**한 흐름으로 설명할 때:**  
**Spec**으로 요구를 정리하고, **Context**를 넣고, **Rules**로 행동을 통제하고, **Skills**로 반복 업무를 재사용한다.  
이 구조로 말하면 요즘 AI 개발 상식을 잘 이해한 인상을 줄 수 있습니다.

*(Skills는 보통 /skills에서 라이브러리처럼 볼 수 있고, 대화에서 "이런 작업용 skill 만들어줘" 식으로 만들 수도 있습니다.)*

## 강의용 요약 (한 장 정리)

1. **Spec-Driven**: 요구를 Spec으로 먼저 정리하고, Spec을 기준으로 개발·검증한다.
2. **Context Engineering**: AI에게 넣을 문맥을 설계해 품질과 일관성을 높인다. (초기엔 프롬프트 엔지니어링이 중요했지만, 요즘은 "어떤 컨텍스트를 주고 어떻게 파악하게 할지"가 더 중요.)
3. **Rules**: AI가 항상 따를 원칙을 정해 두어 결과물 편차를 줄인다.
4. **Skills**: 반복 작업을 표준화해 팀이 재사용할 수 있게 한다.
5. **협업**: 개인 프롬프트가 아니라 **rules/skills로 구조화**해야 팀 자산이 된다.
6. **워크플로우**: Spec 작성 → AI가 TODO/코드 작성 → 개발자는 설계·리뷰에 집중, Spec이 source of truth.

## 다음 단계·심화 학습

강의 후 스스로 파고보기 좋은 방향입니다.

- **Spec 더 다듬기**: OpenAPI/Swagger로 API Spec 작성, BDD 스타일 시나리오(Given-When-Then)를 Spec으로 쓰기.
- **Context 설계**: 어떤 파일을 언제 포함할지(전체 코드베이스 vs 선택 파일), 문서·이슈 링크를 어떻게 넣을지 실험해 보기.
- **평가**: 같은 Spec으로 여러 번 생성해 보며 "Rules를 추가한 뒤 편차가 줄었는지" 비교해 보기.
- **팀 도입**: Rules 1~2개를 팀 규칙으로 정하고, 한 가지 반복 작업을 Skill로 만들어 공유해 보기.
